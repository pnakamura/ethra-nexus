# Phase 15 — Testes Design Spec

**Date:** 2026-04-20
**Status:** Approved
**Scope:** Cobertura de testes unitários e E2E para packages/core, packages/agents, packages/wiki e apps/server

---

## 1. Objetivo

Atingir cobertura de testes >70% em `packages/core`, `packages/agents` e `packages/wiki`, com 6 cenários E2E cobrindo os fluxos críticos do servidor Fastify. Custo zero em CI — LLM sempre mockado em testes automatizados.

---

## 2. Arquitetura — Três Tiers

```
Tier 1 — Puro (zero setup, zero custo)
  packages/core/src/security/validate.ts
  packages/core/src/security/sanitize.ts
  packages/core/src/security/rate-limiter.ts
  packages/agents/src/lib/scheduler/cron-utils.ts

Tier 2 — Mocked (vi.mock para DB + LLM + fetch)
  packages/agents/src/lib/scheduler/event-bus.ts
  packages/agents/src/lib/scheduler/output-dispatcher.ts
  packages/agents/src/lib/db/db-agents.ts  (canExecute, getBudgetAlertsFired)
  packages/wiki/src/embedding.ts
  packages/agents/src/lib/skills/skill-executor.ts  (dispatcher)

Tier 3 — E2E (Fastify inject + PostgreSQL de teste real na VPS)
  apps/server/src/__tests__/e2e/
  LLM mockado via NEXUS_MOCK_LLM=true
  Banco real: DATABASE_URL_TEST
```

**Framework:** Vitest 1.6 (já configurado nos 3 pacotes).
**Localização:** `__tests__/` co-localizado em cada pacote. E2E em `apps/server/src/__tests__/e2e/`.
**CI:** Tier 1 e 2 rodam em todo push. Tier 3 roda com `DATABASE_URL_TEST` definida.
**Custo:** $0 em CI. ~$0.01 por execução de smoke test manual (opcional, não automatizado).

## 3. Estratégia de Mocks

### Tier 1 — Sem mocks (funções puras)

```typescript
import { validateCron, calcNextRun } from '../cron-utils'

test('validateCron rejeita expressão inválida', () => {
  expect(validateCron('não é cron')).toBe(false)
  expect(validateCron('0 9 * * 1-5')).toBe(true)
})
```

### Tier 2 — Três mocks reutilizados

**Mock 1: DB (Drizzle)**
```typescript
vi.mock('@ethra-nexus/db', () => ({
  getDb: () => mockDb,  // objeto com .select/.insert/.update encadeáveis
  agentSchedules: {},
  agentEventSubscriptions: {},
  scheduledResults: {},
  agents: {},
  budgets: {},
  auditLog: {},
  aiosEvents: {},
}))
```

**Mock 2: LLM Registry**
```typescript
vi.mock('../provider', () => ({
  createRegistryFromEnv: () => ({
    complete: vi.fn().mockResolvedValue({
      content: 'resposta mockada',
      input_tokens: 100,
      output_tokens: 50,
      estimated_cost_usd: 0.001,
      provider: 'anthropic',
      model: 'claude-sonnet-4-6',
      is_fallback: false,
    }),
  }),
}))
```

**Mock 3: fetch global (WhatsApp webhook)**
```typescript
global.fetch = vi.fn().mockResolvedValue({ ok: true })
```

### Tier 3 — E2E

Fastify iniciado via `buildApp()` e testado com `app.inject()` — sem HTTP real. Provider registry mockado via `NEXUS_MOCK_LLM=true` detectado no startup. Banco PostgreSQL real via `DATABASE_URL_TEST`.

## 5. Cenários E2E

### 5.1 Execução de skill via API
```
POST /api/v1/aios/execute { agent_id, skill_id: 'wiki:lint' }
→ 200, { answer: '...', tokens_used: N, cost_usd: N }
```

### 5.2 Criação e validação de schedule
```
POST /api/v1/schedules { cron_expression: '* * * * *', ... }
→ 201, next_run_at calculado corretamente

POST /api/v1/schedules { cron_expression: 'inválido' }
→ 400, { error: 'Invalid cron_expression' }
```

### 5.3 Isolamento multi-tenant (RLS)
```
Tenant A cria agente.
Tenant B: GET /api/v1/agents/:id_do_tenant_A
→ 404 (não vaza dado cross-tenant)
```

### 5.4 Budget enforcement
```
Agente com budget_monthly = $0.001 e gasto já no limite.
POST /api/v1/aios/execute
→ 200, { error: { code: 'BUDGET_EXCEEDED' } }
```

### 5.5 Enable/disable de schedule
```
POST /schedules → PATCH /schedules/:id/disable → GET /schedules/:id
→ enabled: false
```

### 5.6 Webhook com secret inválido
```
POST /api/v1/webhooks/:slug/custom-event (sem X-Webhook-Secret) → 401
POST /api/v1/webhooks/:slug/custom-event (secret errado) → 401
POST /api/v1/webhooks/:slug/custom-event (secret correto) → 202
```
